Storage

Two embedded databases, no server. LanceDB holds chunks plus their vectors and BM25 index. Kuzu holds the relationships between them.

LanceDB · chunks + vectors + BM25·Kuzu · graph·both embedded
01 · Why two databases

Different queries, different shapes

Vector and BM25 retrieval are best served by a columnar store with a vector index. Graph traversals (cross-references, "see also" edges, parent/child sections) are best served by a graph database. Trying to do both in one engine costs performance on at least one of them.

So the architecture splits cleanly. LanceDB owns the chunk text, the embedding vectors, the BM25 index, and the per-chunk metadata. Kuzu owns the relationships: which chunks cross-reference which, which definitions an entity resolves to, which prior versions a section supersedes. Each query goes to whichever store fits.

02 · LanceDB

Chunks, vectors, BM25

Every chunk written by the per-agency chunkers lands in a LanceDB table. The text column stores the raw chunk; the vector column stores the 3072-dimensional embedding from text-embedding-3-large; a tantivy-backed BM25 index runs over the text column. Metadata fields (citation, section title, agency, content type, table flag) are searchable and filterable.

Why LanceDBpicked v1
Hybrid in one enginevector + BM25 native
Both retrievers query the same store, returning Arrow batches. No second service to keep in sync. The hybrid lift comes for free at query time.
Embedded · no serverzero ops
Reads and writes go through a Python or Rust process. No daemon, no port, no separate deployment. The data lives next to the code as Lance-format files on disk.
Measured latencyat 89,855 chunks
Real measurements taken 2026-05-17 on the production index: vector top-70 = 6.1 ms p50 / 7.9 ms p95, BM25 (FTS, tantivy) top-70 = 7.2 ms p50 / 8.5 ms p95, vector with agency-filter = 9.0 ms p50. Hybrid fusion adds about 1-2 ms. Raw numbers in docs/explainers/results/storage_stats.json.
Batch inserts criticalingestion gotcha
Inserting chunks one at a time blows up storage size by writing many small Lance fragments. The ingestion pipeline batches into 1000-chunk inserts; this matters more than vector index choice.
03 · Kuzu

The graph layer

Every chunk is also a node in Kuzu. Edges encode the relationships that vector similarity cannot capture: explicit cross-references ("see also SMC 23.54.020"), parent and child sections, prior versions of a renumbered rule, and links from regulatory entities to their canonical definition.

At query time the graph is traversed only after retrieval has narrowed the candidate set to about 5 chunks. Each is expanded by 2 hops, capped to about 5 added nodes per query. The fan-out is bounded; the cost is paid only when graph context actually helps.

Why Kuzupicked v1
Embedded graph DBno server
Same model as LanceDB: a library that opens a database file. No separate process, no clustering setup, no operator overhead.
Fast path queriesmeasured
2-hop expansion across CITES, IMPLEMENTS, DEFINED_BY, SUBJECT_TO, AMENDED_BY measured on the live graph (95,071 nodes, 311,146 edges): p50 = 1.0 ms, p95 = 4.2 ms, max = 6.1 ms over 30 random starting chunks. Independent published benchmarks put Kuzu roughly 300x faster than Neo4j on similar 2-hop workloads.
Vela forkproduction multi-writer
Original Kuzu was archived in late 2025. We use the Vela-Engineering community fork, which adds production multi-writer support and continues active maintenance.
04 · What we rejected

Alternatives and why not

Vector stores considered
PineconeSaaS only
Excellent at scale, but a hosted service with monthly minimums and data egress. Our corpus fits on a laptop. The operational complexity is unjustified at this size.
Weaviateserver required
Strong hybrid search, well-designed schema. But the operational footprint (Docker, persistent volume, schema migrations) is a step backward from embedded.
Chromano native hybrid
Easy to start. Hybrid search requires bolting BM25 on yourself. LanceDB ships hybrid as first-class.
pgvectorPostgres add-on
Fine if you already run Postgres for other reasons. We don't, and adding it just for vectors brings a database we'd otherwise not need.
Neo4jgraph alternative
Industry standard, but heavy at this scale: separate server, JVM, slower path queries than Kuzu in our benchmarks. Defaults to too much for what we need.
