Retrieval

How a question gets turned into a short list of chunks. Hybrid retrieval, reranking, graph expansion, and an entity layer that knows regulatory terms.

vector + BM25 fusion·cross-encoder rerank·2-hop graph expansion
01 · Why retrieval matters

Retrieval decides what the LLM gets to see

The synthesis step is the same model across every arm. The thing that actually differs is what arrives in its context window. A retrieval system that returns the wrong chunk loses before the LLM writes a single token. A system that returns the right chunk plus an extra cross-referenced provision wins the cross-agency questions.

The retrieval pipeline has four stages after the query is parsed: hybrid lexical-and-semantic search, a cross-encoder reranker, graph expansion through the knowledge graph, and a custom regulatory entity layer.

02 · Vector + BM25 fusion

Two retrievers, fused

Dense vector search handles paraphrases. Sparse BM25 handles exact identifiers, numbers, and rare terms. They fail in opposite directions; combining them covers both cases.

The two retrieversRRF fusion
Vector retrievaltop-50 · cosine
Each chunk is pre-embedded with text-embedding-3-large (3072 dim) and indexed in LanceDB. At query time, the query is embedded and the top-50 nearest chunks by cosine similarity move forward. Wins on paraphrased questions: "max height in lowrise" matches "maximum structure height in LR1 zones".
BM25 retrievaltop-20 · keyword
Sparse term matching. Catches exact section identifiers ("SMC 23.54.030"), specific numbers ("13 feet"), and defined terms. Where vector fails: numeric tokens and rare section refs that don't carry strong embeddings. Runs in parallel with vector.
Reciprocal rank fusionhow we combine them
Both candidate sets are merged using RRF: each chunk gets a fused score of sum(1 / (k + rank_i)) across retrievers. Tunable per agency. Default k=60. Yields about 15% lift on regulatory-style questions over either retriever alone.
03 · Reranking

Cross-encoder rerank

The top 70 fused candidates (50 vector + 20 BM25, deduplicated) get scored by a cross-encoder that reads the query and each chunk together. This is a higher-precision signal than cosine similarity. Top 5 survive into the synthesis context.

Why a rerankerprecision step
Bi-encoder limitationvector + BM25 alone
Vector and BM25 both score chunks independently of the query. A chunk that mentions every keyword can score high without containing the answer.
Cross-encoder signalBGE reranker
The reranker sees the query and the chunk together, so it scores joint relevance. A chunk that discusses tandem parking scores below the chunk that defines tandem parking, even if both contain the term.
Where it matters mosttier-5 conflict
On conflict questions, vector returns commentary and rules together. The reranker pushes the actual rule chunks to the top, so the answer cites the rule rather than the commentary about the rule.
04 · Graph expansion

Following cross-references through the graph

Each top-5 chunk has a node in a Kuzu graph database. The graph encodes cross-references ("see also"), parent/child sections, prior versions, and named-entity links. A 2-hop expansion adds 3 to 5 linked nodes that contain the rest of the answer in cross-agency questions.

For Tier-1 single-section lookups this is mostly a no-op. For Tier-3 cross-doc questions it adds the linked agency. For Tier-5 conflict questions it is essential: the graph knows DR 8-2003 cross-references SMC 23.54.020. Without graph expansion, Naive RAG retrieves one side of a conflict and never sees the other.

05 · Entity layer

Custom regulatory entities

A custom named-entity recognizer flags terms that have specific legal meaning in the Seattle regulatory corpus: "tandem parking", "ADU", "shoreline area", "false premise". Each entity resolves to a canonical citation set.

The entity layer also detects gaps. If a question expects to find a graph node for an entity ("vertical clearance for bus space") and the graph has no such node, the system surfaces the silence rather than inventing a number. The same layer detects premise claims ("since X has no Y") and triggers a verification path before synthesis.
