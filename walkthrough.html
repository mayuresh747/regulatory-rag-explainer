02 · chunk

Chunks come out with their citations attached.

The per-agency chunker (regex first, sub-chunk at A/B/C if too long, semchunk fallback) emits sub-chunks. Every chunk carries the citation, section title, content_type, and an is_table flag. The chunk that actually holds the gold answer is highlighted in green.

03 · graph

Each chunk also lives as a node, with typed edges.

The same chunk lives as a Chunk node in Kuzu. The node carries denormalized fields (agency, citation, section_number) so traversals stay cheap. Full text stays in LanceDB. Edges include the parent document (DOC_HAS_CHUNK), the next sub-chunk (NEXT_SECTION), and outward links (SUBJECT_TO, CITES, IMPLEMENTS, AMENDED_BY) with a confidence value attached.

04 · classify

The question gets a tier and an intent.

Before retrieval runs, GPT-4.1-mini classifies the question. The classification is cheap (~1 second) and routes the question into the right retrieval profile and the right synthesis prompt slot. False premises get flagged here and become a correction directive in the prompt.

Classifier output · GPT-4.1-minilatency

The classifier shapes the synthesis prompt. See Synthesis → for the rubric.

05 · retrieve

Three retrievers run. Watch which one wins.

Vector and BM25 run in parallel against LanceDB. Linear fusion (0.55 vector / 0.45 BM25, with min-max normalization) merges them into the candidate list. Graph traversal then runs against Kuzu starting from the fused seeds. The four tabs below cycle automatically when this section comes into view. You can also click a tab to inspect it.

06 · synthesize

GPT-5.1 reads a numbered context with a strict citation rule.

The synthesizer receives the question, the top 15 fused chunks formatted with bracketed citation headers, and a system prompt that demands verbatim citations and a "Not found in provided sources" response when the chunks do not support a claim.
